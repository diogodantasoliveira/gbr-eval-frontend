# gbr-eval-frontend

Admin panel for [gbr-eval](https://github.com/diogodantasoliveira/gbr-eval). Next.js 16 App Router + SQLite (local-first) + Drizzle ORM + shadcn/ui + Tailwind v4.

- 40 pages covering: skills, golden sets, tasks, rubrics, runs, conventions, contracts, calibration, alerts.
- 57 API routes under `/api/*` (auth-protected by default).
- SQLite file at `./gbr-eval.db`, 24 tables, WAL mode.

This repo does **not** stand alone for eval execution — that lives in the backend ([gbr-eval](https://github.com/diogodantasoliveira/gbr-eval)). Clone both side-by-side:

```
your-workspace/
├── gbr-eval/            ← Python CLI + test framework
└── gbr-eval-frontend/   ← this repo
```

## Prerequisites

| Tool | Minimum | Install |
|------|---------|---------|
| Node.js | 20 | https://nodejs.org/ (LTS) |
| pnpm | 9 | `corepack enable pnpm && corepack prepare pnpm@latest --activate` |
| Git | any | system package manager |

For the full stack (backend + frontend + sample data), follow the single-script bootstrap in the backend repo. This README is only for the frontend half.

## One-shot setup (full stack)

If you haven't cloned either repo yet, run the bootstrap from the backend — it clones both, installs everything, and seeds the DB:

```bash
curl -O https://raw.githubusercontent.com/diogodantasoliveira/gbr-eval/main/setup-local.sh
chmod +x setup-local.sh
./setup-local.sh
```

Skip to [Running the dev server](#running-the-dev-server) when that finishes.

## Manual setup (frontend only)

```bash
git clone git@github.com:diogodantasoliveira/gbr-eval-frontend.git
cd gbr-eval-frontend

cp .env.example .env.local       # defaults are fine for local dev
pnpm install                     # native deps compile via pnpm.onlyBuiltDependencies
pnpm db:push                     # applies Drizzle schema → ./gbr-eval.db (24 tables)
pnpm db:seed                     # inserts 5 P0 skills + 40 field schemas
pnpm dev                         # http://localhost:3002
```

## Running the dev server

**Port is fixed at 3002.** Both `dev` and `start` pass `-p 3002` explicitly in `package.json`. See [`PORTS.md`](./PORTS.md).

```bash
pnpm dev          # turbopack dev server, hot reload
pnpm build        # production build
pnpm start        # serve built app on 3002
```

Open **http://localhost:3002**.

## Environment variables

Full reference in [`.env.example`](./.env.example). Key ones:

| Var | Purpose | Local-dev default |
|-----|---------|-------------------|
| `PORT` | Documentation only — the actual port is set in package.json scripts | `3002` |
| `DATABASE_URL` | SQLite file path (relative to repo root) | `./gbr-eval.db` |
| `ADMIN_API_TOKEN` | Required by `src/proxy.ts` to allow any `/api/*` call | unset (bypassed via `DISABLE_AUTH`) |
| `DISABLE_AUTH` | When `true` AND `NODE_ENV=development`, skips `ADMIN_API_TOKEN` check | `true` |
| `RATE_LIMIT_MAX` | Requests per IP per 60 s window | `100` |
| `WEBHOOK_SECRET` | Bearer token for `POST /api/runs/webhook` (CI run ingestion) | generated by `setup-local.sh`; otherwise set with `openssl rand -hex 32` |

**Never commit `.env.local`** — the `.gitignore` protects it. `.env.example` is tracked via `!.env.example` exception.

## Seeding the database

`pnpm db:seed` only inserts the **5 P0 skills and their field schemas**. To get the full "Diogo-state" experience — golden sets, engineering tasks, historical runs, and convention rules — run the backend's sync tool:

```bash
# from ../gbr-eval/, with the frontend dev server running
uv run python tools/sync_frontend.py --base-url http://localhost:3002

# 18 convention rules
curl -X POST http://localhost:3002/api/conventions/import-claude-md \
     -H 'Content-Type: application/json' -d '{}'
```

See the [backend README](https://github.com/diogodantasoliveira/gbr-eval/blob/main/README.md#4-populate-the-rest-of-the-data) for the historical-runs import snippet.

## Webhook ingestion

The backend can POST `EvalRun` JSONs directly after an eval run:

```bash
SECRET=$(grep ^WEBHOOK_SECRET .env.local | cut -d= -f2-)
curl -X POST http://localhost:3002/api/runs/webhook \
     -H "Authorization: Bearer $SECRET" \
     -H 'Content-Type: application/json' \
     --data-binary @../gbr-eval/runs/self_eval_2026_04_19.json
# -> {"run_id":"...","imported":true}   HTTP 201
```

The CI pipeline in `../gbr-eval/.github/workflows/ci.yml` is wired to POST here on every run — set `WEBHOOK_SECRET` and `WEBHOOK_URL` as repo secrets.

## Troubleshooting

**First run of `pnpm install` warns about "ignored build scripts".**
That's only a warning if the allowlist hasn't been set. This repo's `package.json` has `pnpm.onlyBuiltDependencies: ["better-sqlite3", "esbuild", "msw", "sharp", "unrs-resolver"]`, which resolves it automatically. If you see runtime errors about `better_sqlite3.node` missing, fall back to `npm rebuild better-sqlite3`.

**HTTP 500 on every `/api/*` with "Could not parse module '[project]/src/middleware.ts'".**
Stale Turbopack cache from before the Next 16 rename. Kill the dev server, `rm -rf .next`, restart.

**"middleware file convention is deprecated" warning on startup.**
Means `src/middleware.ts` still exists. It was renamed to `src/proxy.ts` in the Next 16 migration — if you see this warning, delete any lingering `middleware.ts`.

**`/api/*` returns 503 "Server authentication not configured".**
`ADMIN_API_TOKEN` is unset AND `DISABLE_AUTH !== "true"`. For local dev, the generated `.env.local` sets `DISABLE_AUTH=true` — ensure the file exists (`cp .env.example .env.local`).

**`POST /api/runs/webhook` returns 503 "Webhook not configured".**
`WEBHOOK_SECRET` is unset. `setup-local.sh` generates one automatically; if you did manual setup, add `WEBHOOK_SECRET=$(openssl rand -hex 32)` to `.env.local`.

**`POST /api/runs/webhook` returns 422 "Validation failed".**
Old eval-run JSONs include `"tier": null` / `"finished_at": null`. The zod schema allows missing fields but not `null`. Strip those keys before POSTing — example in the backend README.

**Dark mode colors look broken.**
Tailwind v4 uses semantic tokens (`bg-background`, `text-foreground`). Never hardcode numeric colors like `bg-gray-100` without a `dark:` variant — the convention is listed in `/conventions` (rule: `dark_mode_variants`).
